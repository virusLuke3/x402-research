# 比赛要求与项目映射

这份文档用于把比赛要求、评审关注点和 AutoScholar 当前版本之间的关系讲清楚，方便写 README、录 demo 和准备答辩。

## 比赛的核心期待

从题目描述来看，评审最看重的不是“又一个 AI 工具”，而是下面这几件事：

- agent 与 agent 之间能否形成协作关系
- 服务能力能否被定价、购买、交付和复用
- x402 是否被用作机器可读的支付挑战与权限释放层
- Stacks 是否真实参与了结算或验证
- 项目是否有足够强的未来感和协议感

## AutoScholar 当前最匹配的点

AutoScholar 目前最强的叙事应该是：

- 一个高价值的 research molbot
- 可以为复杂研究任务报价
- 通过 x402 暴露支付与解锁语义
- 通过 Stacks testnet 完成真实签名与验证
- 在付款后返回可供其他 agent 继续消费的结构化结果

这比“科研搜索工具”这个说法更符合评审语言。

## 与题目的一一对应

### 1. Molbot-to-molbot commerce

对应方式：

- 用 `taskTree` 展示 manager 和 specialist 的任务分解
- 用 `commerceTrace` 展示报价、付款、验证、交付
- 用 `Agent Handoff Packet` 证明结果不是只给人看的

### 2. x402 on Stacks

对应方式：

- 后端返回 x402 风格挑战信息
- 前端触发 Stacks 合约调用
- 后端通过 Hiro 校验交易结果

### 3. Specialized skill molbot

对应方式：

- 把 AutoScholar 定位成 premium research capability
- 强调它输出的是高价值研究资产，而不是普通搜索结果

## 当前版本真实完成了什么

- 前端任务创建与支付流程
- 后端报价、任务树、支付轨迹和输出打包
- `arXiv + OpenReview + 本地框架` 的研究检索与综合
- Stacks testnet 上的真实签名和验证路径
- Markdown + JSON 双形态交付

## 当前版本还没有完全完成什么

- 独立部署的 specialist molbot 之间互相调用与付费
- sBTC / USDCx 的真实链上转账路径
- 注册表、声誉系统和 agent 发现机制

这些内容可以讲成 roadmap，但不要在文档里写成已经完成。

## 文档和演示时应该强调什么

- 不要把项目讲成“买一篇报告”
- 要讲成“购买一个可复用的研究工作流”
- 不要只展示结果页
- 要展示任务树、支付轨迹、结算验证和输出包
- 不要把 Stacks 说成背景板
- 要明确它承担了结算与验证职责

## 推荐的一句话介绍

AutoScholar 是一个基于 x402 和 Stacks 的 research molbot：它可以为复杂研究任务报价、在付款后释放专业能力，并输出可供其他 agent 继续消费的结构化研究资产。

## 推荐答辩策略

如果评审问“为什么这符合题目”，可以直接回答：

1. 它是一个 specialized skill molbot。
2. 它通过 x402 管理付费能力释放。
3. 它在 Stacks 上完成结算验证。
4. 它的输出是给下游 agent 继续使用的，而不只是给人阅读。
